## 语言特点

### 优点

1. **天然支持高并发**  
   通过 goroutine 和 channel，并发模型轻量；goroutine 创建与调度开销远小于操作系统线程，适合高并发场景。

2. **学习成本低**  
   语法相对简洁，没有类继承等一整套 OOP 包袱（早期无泛型，1.18+ 已有泛型），上手与投入生产的曲线较平。

3. **性能与部署**  
   编译为本地机器码，无 JVM 虚拟机层；配合不断优化的 GC，整体执行效率与延迟表现适合多数服务端场景。

4. **单文件部署**  
   静态链接常见，二进制往往可单独分发，依赖少，便于容器与边缘部署。

### 与 Java 的对比（常见说法）

| 维度 | Go | Java |
| --- | --- | --- |
| 语言复杂度 | 语法与特性集相对小 | 语言与生态特性更多 |
| 类型与复用 | 无类继承，用组合、接口表达多态 | 继承 + 接口等多态手段 |
| 并发 | goroutine、channel | Thread、`Runnable`/线程池等 |
| 运行方式 | 编译为机器码，直接进程运行 | 字节码 + JVM |
| 垃圾回收 | 并发式 GC（实现随版本演进） | 多种收集器（如 G1、ZGC 等） |

## 引用语义：哪些类型「像在引用」

Go 规范里**没有「引用类型」的官方清单**，更贴切的说法是：**值里往往带着指针或句柄，赋值/传参时拷贝的是这层「头」，底层数据或运行时对象常被共享**。

- **切片 `[]T`**：头是「指向数组的指针 + len + cap」，多数情况下与赋值来源**共享底层数组**（`append` 扩容后可能不再共享）。
- **`map`、`channel`**：变量里是运行时对象句柄，`m2 := m1` 指向**同一张表 / 同一通道**。
- **`func`**：函数值对应**代码/闭包环境**的引用，拷贝变量不等于复制一份函数体。
- **指针 `*T`**：存的是地址，天然共享一块 `T` 的存储。
- **`interface`（任意接口类型）**：接口值里是类型信息 + 数据字；数据侧经常是**指针**，整体常表现为共享底层。

**小结**：和「纯值类型」相比，这些类型更容易出现**浅拷贝共享**；需要独立副本时要自己做**深拷贝**或不可变设计。

## 深拷贝

在存在共享底层时，若需要互不影响的副本，常见做法：

1. **JSON / 等序列化再反序列化**：通用但性能一般，且要注意字段导出、类型限制等。
2. **`encoding/gob`**：往往比 JSON 快，能覆盖更多 Go 内置类型组合。
3. **第三方库**：按项目选型。
4. **手写拷贝**：结构清晰、字段少时最直接可控。

## 闭包

**闭包** = 函数 + 它所捕获的外层变量环境。

```go
func createCounter() func() int {
	count := 0
	return func() int {
		count++
		return count
	}
}
```

- **用途**：封装状态（类似私有字段）、工厂/生成器、回调里固定当前上下文等。
- **与普通函数**：闭包多了一层**捕获变量**的生命周期，外层局部变量在闭包仍可达时不会按普通栈帧释放；因此内存与并发安全都要额外注意（捕获的变量并发写要加锁或用通道等）。
- **注意**：闭包捕获的是**变量本身**（循环里开 goroutine 时常见踩坑），不是循环当时的值拷贝，除非传入参数或新建局部变量再捕获。

## Gin 框架简介

用 Go 写的轻量 Web 框架，常见卖点：

1. **路由**：基于 HttpRouter 类实现（如 Radix Tree），路由匹配在高压 API 场景下开销较低。
2. **中间件**：统一挂载日志、鉴权、限流、Recovery（捕获 panic）等，与业务 handler 解耦。
3. **参数绑定**：JSON、Query、Form 等到 struct，并可结合 validator 做校验。
4. **Recovery**：避免单个请求 panic 拖垮整进程（仍须正确设计错误处理与资源释放）。
