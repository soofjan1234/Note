## 一、GMP由来 可以画什么

### 1. 传统多线程的问题
- 图类型：问题聚类图（Relationship）。
- 画面元素：
  - 中间一个主框：`传统多线程`
  - 四周四个问题框并连线：
    1. 线程内存开销高（常见 1~8MB）
    2. 上下文切换成本高（寄存器/PC/栈切换）
    3. 创建销毁依赖内核调用
    4. 高并发下调度压力变大
- 目的：先让读者直观理解“为什么线程模型在高并发下会重”。

### 2. 单线程用户态调度
- 图类型：对比图（Comparison）。
- 画面元素：
  - 左侧：`内核线程直接调度`（画“内核边界”提示进入内核）。
  - 右侧：`M + G 用户态切换`（一个 M 轮流执行多个 G）。
  - 在右侧加一个“全局队列 + 全局锁 schedlock”小区域，提示后续瓶颈来源。
- 目的：说明 GM 为什么先解决了“切换重”，但仍埋下“全局锁竞争”的问题。

### 3. 多线程调度器的问题
- 图类型：关系图（Relationship）。
- 画面元素：
  - 画多个 `M` 同时连到一个“全局队列 + schedlock”中心节点。
  - 旁边放三个问题卡片：
    1. 锁竞争加剧（多个 M 抢同一把锁）
    2. 跨线程任务交接成本高
    3. 缓存局部性差（G 换 M 后缓存变冷）
    4. `G syscall 阻塞可能拖住 M`。
- 目的：承接到“为什么必须引入 P”。


### 4. GMP + 工作窃取机制图
- 图类型：关系图 + 流程箭头。
- 画面元素：
  - 两个 P（`P1`、`P2`），每个都有本地 runq。
  - `M1` 绑定 `P1`，`M2` 绑定 `P2`。
  - 画 `P1` 很忙、`P2` 空闲时，从 `P1` 队尾向 `P2` 画“偷任务”箭头（标注：偷一半，至少 1 个）。
  - 旁边写：本地优先，空了再看全局。
- 目的：让“窃取均衡”一眼能懂。

### 5. 抢占机制对比图（协作式 vs 信号式）
- 图类型：左右对比图（Comparison）。
- 左侧（协作式抢占，Go 1.14 前）：
  - 主路径：`运行过久` -> `打标记 stackPreempt` -> `等函数调用点 stack check` -> `主动让出`
  - 角落标注：`for {}` 这类无函数调用场景不容易及时让出。
- 右侧（信号抢占，Go 1.14 起）：
  - 主路径：`sysmon 发现超时` -> `向 M 发信号` -> `内核打断` -> `强制进入抢占路径`
  - 补一句：不再完全依赖 G 自觉。
- 目的：说明“公平性增强”的关键来自信号抢占。

## 二、GMP机制 可以画什么

### 1. G/M/P/schedt 关系总览图（主图）
- 图类型：层级图 + 关系图。
- 画面元素：
  - 顶部：`schedt`（全局调度中心）
  - 中间：多个 `P`（每个 P 画 `runnext` + `runq` + `gFree`）
  - 下层：`M` 绑定 `P`，`M` 执行 `G`
  - 旁边备注：`M 必须绑定 P 才能执行 G`
- 目的：建立“谁管谁、谁依赖谁”的总体心智图。

### 2. 找 G 顺序流程图（findRunnable 大白话版）
- 图类型：流程图（Flowchart）。
- 顺序节点按文章写法：
  1. 特殊任务（trace / GC worker）
  2. 公平性检查（周期看全局）
  3. 本地队列（先 `runnext` 再 `runq`）
  4. 全局队列
  5. netpoll
  6. stealWork
  7. 仍无任务 -> 让出 P / 自旋 / 休眠
- 目的：读者看完就知道“调度器不是乱找活”。

### 3. 阻塞与恢复生命周期图
- 图类型：状态流转图。
- 画面建议：拆成上下两条泳道，分别画“普通阻塞”和“syscall 阻塞”。
- 上泳道（普通阻塞）状态节点：
  - `running` -> `waiting` -> `runnable` -> `running`
  - 触发标注：channel 不匹配、锁拿不到、sleep、网络等待。
  - 恢复标注：条件满足 / 资源就绪 / 被唤醒。
- 下泳道（syscall 阻塞）状态节点：
  - `running` -> `syscall` -> `runnable` -> `running`
  - 过程标注：G 进 syscall，M 可能被挂住，P 可被接管。
  - 恢复标注：优先尝试拿回 P；拿不到则先入全局队列再等待调度。
- 目的：强调“阻塞分两类，恢复路径也不同”，避免把 syscall 和普通 waiting 混成一类。

## GMP.抢占机制对比图

### 5.1 这张图怎么画更稳（按 skill 落地）

- 图类型：`Comparison`（左右双栏）
- 画布建议：`1100 x 760`，左右留白各 `70px`，中间分隔线 `x=550`
- 字体：统一 `fontFamily: 5`，正文 `16-18px`，标题 `24px`，副标题 `18px`
- 颜色建议（同色系描边）：
  - 协作式列：浅橙底 `#ffd8a8` + 深橙描边 `#c2410c`
  - 信号式列：浅蓝底 `#a5d8ff` + 深蓝描边 `#1e40af`
  - 问题/风险卡：浅红底 `#ffc9c9` + 深红描边 `#b91c1c`
- 箭头规则：主路径用实线，补充说明用虚线；避免箭头穿过文本框中心，优先边缘连接。

### 5.5 不画卡片版：单 G 时间轴对比（推荐）

如果你不想画“节点卡片流程图”，可以改成**同一条时间轴上的执行片段对比**，更像 runtime 真实发生的事。

#### 画面结构

- 左半：`协作式（Go 1.14 前）`
- 右半：`信号式（Go 1.14 起）`
- 两半都只画一个 `G1` + 一个 `M1`，避免信息噪音

#### 左半（协作式）怎么画

1. 画一条水平时间轴：`t0 -> t+10ms -> t+...`
2. 在时间轴上放两个“函数片段”长条：
   - `func A`（正在跑）
   - `func B`（下一个函数调用点）
3. 在 `t+10ms` 位置画一个小标记：`stackguard0 = stackPreempt（已打标）`
4. 用一条较长箭头从 `t+10ms` 指到 `func B` 开始处，标注：
   - `等待到下一个函数调用点才触发 stack check`
5. 在底部加一句风险说明：
   - `若是 for {} 无调用点，这段等待会很长`

#### 右半（信号式）怎么画

1. 同样画一条时间轴：`t0 -> t+10ms -> t+10ms+ε`
2. 画一个连续执行条：`G1 on M1`
3. 在 `t+10ms` 放事件点：`sysmon 发现超时`
4. 立刻接一个短箭头到 `SIGURG` 事件，再接到：
   - `内核中断 -> sighandler -> asyncPreempt -> mcall/g0 -> gopreempt_m`
5. 最后落点写：
   - `G1 回可运行队列，M1 可去执行其他 G`

#### 视觉细节（让图一眼懂）

- “等待”用**长箭头**（浅橙系），“中断”用**短且陡的箭头**（浅蓝系）
- 左半可用虚线边框强调“依赖检查点”，右半用实线边框强调“强制介入”
- 底部只留一行结论：
  - `协作式：能否及时让出取决于调用点；信号式：调度器可直接打断。`


## GMP机制.GMP关系总览

### 1. 主图结构（推荐改成三层）

`Hierarchy + Relationship` 混合图，三层从上到下：

1. 顶层：`schedt（全局）`
2. 中层：`P1 / P2 / ...`（每个 P 展示本地队列）
3. 底层：`M1 / M2 / ...` 与 `G`（M 执行 G）

### 2. 每层应该放什么

#### 顶层：schedt（总控室）
- `global runq`
- `idle M list`
- `idle P list`
- `global gFree / sudog / defer`

#### 中层：P（调度资源持有者）
- `runnext`（优先位）
- `runq`（本地队列）
- `gFree`（可复用 G）

#### 底层：M + G（实际执行）
- `M`：OS 线程，持有 `g0` 与 `curg`
- `G`：goroutine 执行快照（栈 / PC / SP / 状态）

### 3. 关键连线（避免信息太散）

只保留 4 类箭头，保证一眼可读：

1. `schedt -> P`：全局下发/回收调度资源（实线）
2. `P <-> M`：`M 必须绑定 P 才能执行 G`（粗实线）
3. `M -> G`：执行关系（实线）
4. `P -> schedt`：本地无活时回看全局（虚线）

### 4. 图内文案（控制在一句话）

放在图标题下方：

- `P 持有队列与资源，M 负责执行，G 是被调度的任务，schedt 负责全局协调。`
