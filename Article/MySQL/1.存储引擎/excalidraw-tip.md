## 一、 1.2 InnoDB核心组件与运行流程 可以画什么

### 1. InnoDB内存与后台线程总览图（核心主图）
- **图类型**：层级图 + 关系图（Hierarchy / Relationship）。
- **画面元素**：
  - **内存组件（上半部分）**：一个大框包含 `Buffer Pool (缓冲池)`。
    - 内部画三个小排队的方块链表：`Free List(空位)`、`LRU List(冷热链表)`（中间切分开 `Midpoint`）、`Flush List(脏页链表)`。
    - 旁边放置独立框：`Redo Log Buffer(重做日志缓存)` 和 `额外内存池`。
  - **后台线程（中间协调器）**：一排干活的圆角机器人或齿轮：
    - `Master Thread`（大齿轮，统筹调度）
    - `IO Thread`（处理底层异步回调）
    - `Purge Thread`（清洁回收 Undo）
    - `Page Cleaner`（专门负责刷脏页）
  - **磁盘文件（下半部分）**：画磁盘柱面图标：`.ibd 数据文件`，`Redo Log 日志文件`，`Undo Log`。
- **作用（箭头连线标注）**：
  - 内存通过“后台线程”批量/异步地搬移到磁盘。
  - `Flush List` -> `Page Cleaner` -> 刷入磁盘 `.ibd`。
  - `Redo Log Buffer` -> `Master Thread` -> 刷入重做日志。
- **目的**：整体把握引擎的全貌流程，一眼明白数据页修改与落磁盘的路径分离。

### 2. LRU冷热链表淘汰机制
- **图类型**：流程图 + 状态流转。
- **画面元素**：
  - 画出一条水平链条（带刻度），切分为左侧段（`热区 New 5/8`）和右侧段（`冷区 Old 3/8`）。
  - 交界处插一道旗帜：`Midpoint (观察期入口)`。
  - **读盘新入**的箭头直接指向 `Midpoint`，并打上标注「新读入数据等待 1 秒」。
  - **成功存活且再次被访问**，画实线箭头飞跃到左侧「热区头部」，标注「晋升防污染」。
  - **未被访问**或只是单次全表扫描产生的页面，画曲线箭头流向右侧「冷区尾部」，标注「快速淘汰」。
- **目的**：说明之所以要在 LRU 中分出「冷热」，是为了防止简单的全表扫描把真正的热点缓存挤出内存。


### 3. Checkpoint 机制触发图
- **图类型**：关系图（Relationship / 触发器）。
- **画面元素**：
  - 中心是一个显眼的枢纽 `Checkpoint (刷脏管家)`。
  - 周围画三条汇聚的触发条件（箭头连向中心）：
    1. **内存告急**：`Free List` 见底，踢走 LRU 脏页前触发“小规模刷脏急救”。
    2. **日志告急**：`Redo Log` 循环写满追尾了，必须“强行刷脏”推进 LSN，否则全系统停工卡死。
    3. **例行维护/崩溃恢复**：`Master Thread` 每秒/每10秒平滑刷脏；或者重启时靠 LSN 缩短恢复期。
- **目的**：说明既然有内存缓存，这就必须有一个机制来主导“什么时候把脏东西真正倒进磁盘”，并防范最坏情况。

## 二、 1.3 InnoDB关键特性原理解析 可以画什么

### 1. Change Buffer 工作流转图
- **图类型**：流程图（Flowchart）。
- **画面元素**：
  - 先设置一个卡口：菱形判定 `是唯一索引吗？且在不在内存里？`。如果都是“否”，进入主角路径。
  - **写缓存阶段**：生成修改事件，短箭头送进 `Change Buffer (物理存在于共享表空间的 B+树)`，并提前返回向客户端汇报完成。
  - **被动/主动合并 (Merge) 阶段**（双管齐下）：
    - 触发1：之后某次普通查询碰巧把所在的二级索引页读进内存了。
    - 触发2：到了 Master Thread 例行苏醒时间 / 数据库快关机了。
    - 最终两者合二为一，指向原真实索引页：“一次性大范围合并”。
- **目的**：清晰解释它拦截“非唯一”随机磁盘写，并化零为整的效能魔法。

### 2. Doublewrite（两次写）自救架构图
- **图类型**：流程图 + 对比强调。
- **画面元素**：
  - **起因卡片（大红字）**：标出 `16KB 内存脏页` 遇到系统 `4KB 原子写` 半途断电，引出致命伤：“页断裂 (内部数据残缺，连 redolog 都无法复原)”。
  - **双重落盘路线（一慢一快）**：
    1. 脏页离开缓冲池，进入内存专属区：`Doublewrite Buffer`。
    2. 第一写（大粗箭头，标注“顺序写备胎，贼快”）：写入磁盘**共享表空间**的连续块做底层备份。
    3. 第二写（多条分叉细线，标注“离散写正身”）：分别找到对应的各表的 `.ibd` 真文件位置。
  - **自救虚线（生死分叉）**：
    - **情况 A (虚惊一场)**：如果在步骤 2（写备胎时）断电，说明原 `.ibd` 里的正身根本还没被碰过，数据没毁容，重启后直接正常走 `Redo Log` 重做即可。
    - **情况 B (真遇险了)**：如果在步骤 3（离散写正身时）断电导致页损坏，画一条粗虚线箭头从 **共享表空间备胎** 里捞出这个完好的快照页，覆盖损坏页，再用 `Redo Log` 补齐差额找回最新状态。
- **目的**：着重展示 “虽然叫了两次写，但前头是极快的顺序追加打底，不损耗多少性能却换来了极限的页损毁修复力”。


### 3. AHI / AIO / 刷新邻接页 (辅助特性对比卡片)
这三个特性比较独立，可以画作“一图三卡”的综合展示牌：
- **AHI (自适应哈希索引) 卡片**：
  - 画 `B+树`(4层高，需要做4次逻辑跳转) 对比 `Hash表`(一步直达)。
  - 标注：“InnoDB 监控爆款热点数据，自动在内存建 Hash，把 O(N) 搜寻转为 O(1)”，旁边用小字提醒：*仅支持等值查询，不支持范围*。
- **AIO (异步 IO) 卡片**：
  - 画多个零散的小读写请求块，经过一个漏斗 (`AIO Merge`)，变成一个巨大的连续物理批处理块交到底层磁盘。
  - 标注：“主线程不仅不阻塞等待，更能将零散 IO 化零为整 (IO Merge)”。
- **刷新邻接页 卡片**：
  - 画一块磁盘区域，标出目标脏页和紧挨着的“邻接脏页”。
  - 左半画 `机械硬盘 HDD`，配个笑脸符号（“磁头转到了就顺便一起刷了，拯救物理机械盘的神技”）。
  - 右半画 `纯固态硬盘 SSD`，配个叉号或警示牌（“SSD 随机极强且寻道为 0，连累未熟透的脏页徒增写压力，建议关闭”）。
