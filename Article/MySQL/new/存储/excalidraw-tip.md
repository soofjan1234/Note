# InnoDB存储 章节配图提示词

遵照 excalidraw-diagram skill。以下用于 `Article/MySQL/new/存储`，配套 `2.1-物理文件结构.md` 和 `2.2-表与索引的行格式.md`；侧重于展示**空间包含关系**与**二进制字段排布**。

## 一、 2.1 物理文件结构 可以画什么

### 1. InnoDB 逻辑存储层级关系图
- **图类型**：层级图 (Hierarchy / Container)。
- **画面元素**：
  - **最外层大框**：`Tablespace (表空间 / .ibd 文件)`。
  - **内部第二层**：划分出 `Segment (段)`：`数据段 (叶子节点)` 、`索引段 (非叶子节点)` 和 `回滚段 (Undo Segment)`。
  - **内部第三层**：在段内画出一组 `Extent (区 / 1MB)`，标注“由 64 个连续页组成”。
  - **内部第四层**：在区内画出若干个小方块 `Page (页 / 16KB)`，并在其中一个页里点出几条线代表 `Row (行)`。
- **目的**：宏观展示 InnoDB 是如何为了优化顺序 I/O，将数据一层层精细管理的。

### 2. 数据页 (Page) 内部解剖图
- **图类型**：关系图 / 结构图 (Structure)。
- **画面元素**：
  - 画一个长方形代表 **16KB 数据页**，从上到下划分为 7 个横条：
    1. `File Header (38B)`：画出指向“前页”和“后页”的双向箭头。
    2. `Page Header (56B)`：记录状态。
    3. `Infimum + Supremum`：画两个小保镖，中间夹着数据。
    4. `User Records`：重点区域，画几行数据并用单向箭头串联。
    5. `Free Space`：留白区域，标注“待分配空间”。
    6. `Page Directory`：在底部画出几个 `Slot (槽)`，箭头指向上方的记录组，标注“二分查找辅助”。
    7. `File Trailer (8B)`：底部的校验位。
- **目的**：透视页内的精密构造，解释为什么 16KB 能跑出高性能。

## 二、 2.2 行格式 可以画什么

### 1. Compact 行格式解剖图
- **图类型**：结构排布图 (Binary Layout)。
- **画面元素**：
  - 一行横向的格子，从左到右依次是：
    - `变长字段长度列表` (紫色，标注“逆序存放”)。
    - `NULL 值列表` (蓝色，标注“1bit 代表一列”)。
    - `记录头 (5Byte)` (橙色，包含 `deleted_flag` 和 `next_record` 指针)。
    - `真实数据 / 隐藏列` (绿色，点出 `Trx_ID`, `Roll_Ptr` 等关键隐藏字段)。
- **目的**：直观展示 InnoDB “能省则省”的设计哲学。

### 2. 行溢出 (Row Overflow) 处理机制对比
- **图类型**：对比图 (Comparison)。
- **画面元素**：
  - 分为左右两部分：
    - **左侧 (Compact 格式)**：画一个数据页，包含一个“大胖子”行，行内留了一截内容（768 字节），后面拖着长长的虚线指向另一个完全独立的 `溢出页`。标注：“藕断丝连”。
    - **右侧 (Dynamic 格式)**：数据页内只剩下一个闪亮的 `Pointer (指针)`，所有内容都飞向了外部的 `溢出页` 链表。标注：“完全甩锅，B+树极速瘦身”。
- **目的**：解释为什么要升级到 Dynamic 格式，以及它如何缓解“页分裂”压力。
