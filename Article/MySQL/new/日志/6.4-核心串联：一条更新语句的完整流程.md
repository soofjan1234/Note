# 6.4 核心串联：一条更新语句的数据落盘生命大图

在这个终点，我们将卸下所有的模块隔离护盾。前面你看到了拆解了各司其职极其复杂的轮子，现在我们要强行把所有齿轮死死铆接咬合在整体战车上，这就是在面试和实践中考验架构全局观最残酷的地狱级修罗场考题——**当我们按下 Update 开始修库到真正确认事务成功中间到底在全貌空间经历了多少折磨**。

## 一、 当一条 Update 飞来之前的血肉冲突与写入（内存战争）

首先你向数据库提交要针对一条内存进行极难更改决战操作 `Update account money = 无穷大 where ID = 我自己`：
1. **落内存改账**：如果在核心大厅（Buffer Pool）找不到对应的页，大军必须调派磁盘抓取出来（顺便走过 Change Buffer 探探路等各种魔法加持，如果需要的话可以一齐进行融合减少搜索次数），总之内存里现在它变成了不可逆极速脏水数据页（脏页）。
2. **在生死记录签下血誓 (Redo Log 生成)**：虽然硬盘大本表还没收到结果，但已经光速把内存里改动的细碎残渣轨迹顺手刻画录入了极速响应区域 `Redo Log Buffer` 之中。

这看似万事俱备，其实危机四伏。如果你现在对所有机器宣告大功告成（直接 Commit），如果电断了或者集群出岔子。主机器（引擎内部靠 Redo 活捉）和外机器从机（Server 外层靠 Binlog 同步存活），一定会彻底失联错频因为双方不同步（谁先写谁后写只要出错断在那都将面临无法抹平的数据黑幕绝境隔离。

这必须通过跨层重载引入终极分布式大同步架构。

## 二、 极限操作：二阶段提交（2PC 宏大交涉史）

要让代表内核硬底子的 **Redo** 军团与包举环宇统帅主从宏大叙事层级外的 **Binlog** 进行互相绝不妥协的拉扯和妥协，就必须牺牲一定量极短性能采用分布协同——**两段式死磕握手（Two-Phase Commit）**：

1. **第一阶段 (Prepare 底气初探准备阶)**：引擎开始把所有的重做生命线（Redo Log）甩落硬塞给磁盘系统深处，但并不敢真正敲锣定音让它立即生效而是给其强加上一层带有待交涉意味诡异的特殊血手印标签：`状态改为 prepare 状态！暂缓最后断定`。（这步说明内核硬盘底子没毛病做好了）
2. **第二阶段决着一：抛出通牒 (Binlog 落盘)**：Server 服务中枢此时接到汇报立刻疯狂全线发动作业：好，把你这次怎么篡改那小子的日志书，全部无情写抛记录倒在那堆 `binlog 备份大录本文件中保存妥当！`。
3. **第二阶段决着二：尘埃落终极落户 (Commit 落锤阶)**：等上述 Server 从外面满含信心告知成功时，原本引擎层苦苦维持压在悬崖底不敢造次的那个底层 Redo 生命记录的标注立刻顺势切过将最终结果撕下：**彻底将其烙成绝对铁印不可动摇（状态修改：commit)！** 全盘事务最终盖棺落定。

**如果突发暴乱断电时的恢复机制验证查实极强逻辑闭环如何展开？**
开机恢复时系统会严丝合缝用这套流程锁死破除质疑。它拿着引擎重做的文件看！
- **Case 1**：状态一检查是 `commit` 落槌：极好不用怀疑说明天时地利人和顺畅！原封还原恢复。
- **Case 2**：状态一检查竟然只维持惨淡挂在极其不上的 `prepare` 半中央状态。
   - 这就面临严酷去查询另一头的对家大书房（去翻 Server 大本营 Binlog），**看看在当时惨案中他们有没有把事最终记入了名流本！**
   - **如果 Binlog 记录存在妥帖**：代表在跨越第三步的超细微最后一丝间隙惨遭雷劈但事实从库早有了同步痕迹底子！必须保真把状态强制补齐挽救这趟这事**拉成 commit 无忧通过。**
   - **如果连对家 Binlog 甚至查无此段失踪**：代表惨死发生在极初的第二阶段还未来得及抛发给天下通知，那就把这个只有底子在苦撑的孤勇残次段在硬盘直接倒退连根废除：**极其残暴地执行 ROLLBACK 回滚打散掉它。** 数据再度平衡拉平没有留下遗憾污点破绽无任何出岔乱序。

## 三、 不屈地向物理特性榨取极限

在上述那套完美保全方案推进之后，由于频次极致压迫硬盘引发狂怒缓慢。由于上述一阶段和二阶段都是强行往极端需要 I/O 硬件狂扫，所以产生两个最顶级的极致压榨机制破局：

1. **宏观层突围：组提交（Group Commit 集束破壁法）**。如果每人完成一次事务都去极高开销向核心盘 `fsync` 冲刷通知将把服务器卡顿陷入泥沼。为了提升大流量的吞吐利用并列等待积攒！把由于这极其相近数微秒的在苦等的几个十个在 `prepare` 等待处理响应提交的几十号事务一把统归！利用一次超巨大的批量洪流直接推入底层落盘！不仅通过减幅压强解决了多进程由于等锁落盘极强塞车的堵塞问题还能借并流吞吐提升一万点并列伤害极大幅优化高并发性能（这绝对是性能翻覆的最精巧神器之一）。
2. **终极数据落底：回到原脏水页（Buffer Pool -> Doublewrite -> .ibd 文件）**
最后不要忘了，所有的这些天女散花的满天日志（Redo/Binlog）和惊呼跌宕全只是极短暂地通知预备安全后手（WAL 的最高旨意），真理的那庞杂臃肿数据还在核心大厅（Buffer Pool 缓存里的一个大脏水块）中待着！他们最后将由单独极其庞大安静沉稳的 `Page Cleaner Thrard 后台老清洁工线程` 通过我们在上篇介绍那套最沉重不凡繁锁的步骤 `第一步放入 Doublewrite 内存池、然后快写落到共享存储区打个备用大印版、最终才一点点缓慢离散滴流倒进真正的硬盘库本表文件归去来兮。`

只有当你走完读懂这最后这幅集齐所有篇幅力量的全盘合图，才算真正触碰摸底领略理解了全域全貌整个浩瀚引擎运转深远壮美！
