# 6.3 Binlog 与主从复制

在这个系统里我们似乎已经集齐了一切。但对于真正要构建大规模可用、动辄容灾切机机房的企业级环境来说。一切单机层面的内敛保平安仍然远远达不到要求。为了打破单点物理限制的僵局扩散全网数据血脉——引入了超强外扩组件：**二进制主控统帅 Binlog。**

## 一、 它到底是什么异类？

需要极其严肃指出的是：**Binlog 它根本不属于底部的 InnoDB 存储引擎。由于地位超然卓越极高，它位于顶部的系统管理层 Server 层！它被几乎所有的其它子级引擎共用！**
它是一本记录那些确确实实执行过更改数据的动作总集。因为极度完整还原当时操作事迹和逻辑，它在“通过重播（Replay）重建恢复某一时点误杀丢失库表以及把数据血气发给千万里外的**从机同步主从库（Replication）功能构建系统集群方面**”大杀四方没有任何取代空间。

### 这张兵书（记录格式）的三大模式演化：
1. **STATEMENT（逻辑流）模式：非常老迈**
   极其简练、它就把你发送的那条 SQL `DELETE FROM WHERE A=B` 老老实实原封存录。带宽容量狂省。但容易造成致命的异星偏差由于有些例如 `NOW()` 是依赖时间乱数生成的、由于重放同步到几万公里外其他时区从机完全产生不一致毁容灾难。所以容易引起分歧不均报错！
2. **ROW（重放实体）模式：铁骑纪律严苛（目前世界最主流）**
   不管多啰嗦复杂的变更计算。它全部记录底层真正那一万条死人被抹杀替换变动之前长成什么状态，一万条之后的最终形态。（记录物理更改变迁前后），无论到天涯海角由于它非常死板硬照绝对极安全地不偏毫秒还原出所有，缺点就是非常容易占据恐怖容量。
3. **MIXED（融合）模式：见机行事**
   基于容量过大致死所以 Server 控制进行分析调和。如果没有那些随机变数坑，就非常轻松用 STATEMENT；如果发现你包含极高危险会发疯重算的随机特性元素就立刻转以铁骑的 ROW 严密封住记载以防跑偏毁库。

## 二、 两对冤家：Redo Log 和 Binlog 的终极碰撞区别

你一定很容易将它俩搞混（其实在实际工程这往往也是灾难易爆雷的考核区域因为太难以归纳：

| 决战核心辨识维度 | Redo Log（血性引擎重做自救军） | Binlog（服务器通用的兵符记录官） |
| --- | --- | --- |
| **归属层级门类** | **InnoDB 内部引擎层**（专享，其他引擎无） | 宏大且所有引擎共用的顶层 **Server 层** |
| **最高防范目标导向** | 全副身家专注在遇到不可抗**宕机崩溃的物理修复 (Crash Recovery)** | 主要在防范跨越重重山海进行**主从同步克隆备份和历史极长审计恢复 (Point-in-time Recovery)** |
| **记录写入结构内容** | 一刀入血的极为纯粹的字节修改定位偏移录：**极硬核的物理日志区块体系。** | 更偏向原汁原味的逻辑行为传达和整行抹除内容体现。逻辑化极高的载体。 |
| **持久留痕的方式特征** | 只留前置的生命极限冲刺圈套环 **采用的是满掉覆盖抹除式流循写入** | 老老实实的流水年鉴库库归档式无情追加 **永不停歇持续新增追加并可根据切割大部头打包冷备绝不循环删抹。** |
| **出列投入磁盘的时机** | 这是千变万化防跌宕重做系统非常**多元在运行态分段偷偷丢在硬盘里** | 严谨地只有在宣判最后彻底干结定音时刻**也就是事务的最后极提交闭幕才统一狂泻抛出**。 |

正因为这两个流派分别有着不同的防灾范围保护和不同的触发写入时间时机，如果遇到半夜惊魂拉闸，这二龙很容易由于一方没落地走偏，这会导致你自己的盘复活了（主库），但从库同步没过去（分库死透拉脱不平记录）。这不得不由下一章去串联化解！
