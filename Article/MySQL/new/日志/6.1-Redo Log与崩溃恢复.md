# 6.1 Redo Log 与崩溃恢复

在这个凡事看硬件、看断电的时代，既然想要快，又要防止中途翻车造成无可挽回的业务雪崩，就必须依靠数据库引擎最强有力的一位硬汉——**Redo Log（重做日志）**。

## 一、 核心底层设计

由于它是为了极致兜底“持久化”（Durability）而存在的物理日志，它的一切构造就是怎么坚韧可靠怎么来：

- **物理存储形态**：它默认不是单一文件，而是以 `ib_logfile0`、`ib_logfile1` 为首组成的极具安全机制的日志组（循环覆盖式系统跑道）。
- **极小颗粒的记录形式（Log Block）**：它是记录物理修改的，这与 Binlog 基于 SQL 或者行逻辑内容完全不是一个次元。它的存储单元默认被框死在 512 字节的 Block 里。原因很简单粗暴——512刚好也是普通机械硬盘的物理底层最小单元。一次完美对齐扇区的大扇面写入，不仅是速度的升华而且彻底隔绝部分损毁无法修复的原子性灾难。
- 里面的记录大概就像天书：“警告：在第 10 空间号码，翻到第 45 页，走 120 步的那个坑里，立刻盖上一层 0xAAAA 字节。无论对错，死命盖”。不需要考虑高深复杂的 SQL 怎么转行分析语法，所以**奇快无比，恢复零开销。**

## 二、 两个永远奔跑的指针：Write Pos 与 Checkpoint

在这条无尽循环奔跑的日志组跑道里，有两个核心角色在上演猫鼠游戏：

1. **Write Pos（当前冲刺指针）**：代表写日志最前端的新箭头。随着每一次发生数据污染修改它就往高处挺入，狂飙推进。
2. **Checkpoint（检查点指针）**：像一个缓慢有气无力拖在后面的重型推土车。代表它前面的所有由于曾经的疯狂修改而弄脏的内存叶数据，已经慢条斯理地**真正写入到了安全不可灭的 .ibd 实体硬盘文件内。**

**安全抹除逻辑**：
由于 Checkpoint 走过的地方意味着那一段历史使命已安全下达磁盘。此时那段遗留在 Redo Log 跑道里的记录就可以毫无负担的作为空场让 Write Pos 下一圈套圈重新覆盖。然而，在系统爆满的极端重载下，当新锐小将 `Write pos` 跑着跑着猛然撞上前排推延慢吞吞车尾 `Checkpoint`，那就是数据库死机崩坏的时候了——它不得不立刻罢工停止全体交易运转接驳，开始在内存大量刷新落脏水，以便移动老油条推土车腾出空间。

## 三、 WAL 的奥义：凭啥先写它就是快？

所有人都在疑惑，既然你最后迟早要把结果塞回硬盘的大铁块（`.ibd` 文件）里，怎么还要吃力不讨好第一环强逼自己写一条 `ib_logfile`？不都是存图硬盘，能节省到哪里去？这叫 **WAL技术 (Write-Ahead Logging 预写防灾技术)。** 这代表着人类压榨机械臂寻道物理特性的巅峰智慧。

- **存原数据由于散乱是绝望的机械苦役**：真实的记录更改散漫分布在庞大表上成百上千个不相关角落里，如果当场全部去把这四百行更新塞回原数据位，磁针手臂会跳跃寻找 400 次随机空档点，开销足以报废数据库所有的性能。这叫极端高昂的**随机 I/O 操作**。
- **存这本流水账却是丝滑的溜冰**：我不用管你究竟影响世界角落哪一行的谁，统一按发生的日志序号向着这个流水记录尾巴（Redo Log 底端）无脑**追加**。写头在固定的那个盘道上顺流滑动丝毫不受干扰，疯狂吞食几万单笔。这叫丝滑快感的高贵**顺序 I/O 操作**。

## 四、 宕机之后的奇迹时刻（崩溃恢复）

万一这台无情奔跑服务器忽然被老板一巴掌踢开电闸掉电到底应该怎么擦补回来？其实就靠着日志量序号。

它是利用了 **LSN (Log Sequence Number，极大的增值自号数)** 的比对复生技术：
1. 一开机它不问表的情况。第一时间疯狂把重伤数据文件页和这根救命稻草流水线对比。
2. 比如当时那一行数据页的头部被雷劈前只更新记录到了 LSN = 800。但在顺着跑道的流水记录赫然留着“这曾经有个 LSN=1000 的高台跃迁”！意味着那遗失漂浮在电脉冲内、还未来得及推移落实的 200 个字节脏魂丢在了深层位面。
3. 它直接通过流水线原封不动复刻、生搬硬套（重做还原）。这叫 **Crash Recovery**，数据库完好的大厦依然矗立不倒。
