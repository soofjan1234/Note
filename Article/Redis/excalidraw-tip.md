### 1. SDS vs C 原生字符串 (对比图)
- **图类型**：上下两个长条对比。

### 2. 从 ZipList 到 ListPack (流程图/演化图)
- **图类型**：展示连锁更新 (Cascading Update) 的痛点。

### 3.1 跳表 SkipList：静态巨无霸微观结构 (基础体系图)
- **图类型**：基础硬件大盘点。
- **绘图细节**：

### 3.2 跳表 SkipList：【查】高空降落伞轨迹 (寻址流向图)
- **图类型**：专攻查询路径与进阶算术的模型图。
- **绘图细节**：
    - **查询流星**：画一条从顶级左侧出发的粗壮红折线：高点向右平移飞跃 -> 撞阻挡物（分值超了）垂直往下掉 -> 再向右找人。

### 3.3 跳表 SkipList：【增】抛硬币晋升机制 (随机拔高动画)
- **图类型**：正在执行插入的一格动画定格。
- **绘图细节**：
    - 在最底层安插一个醒目的黄色 `New Node`。
    - 它头顶画一个带 “Coin Face” 的小抛硬币图示，用冲天光束往上引，旁边大字注明「运气晋升律：抛起 1 枚硬币，是正面就长高一层，直到抛出反面定格当前总高」。
    - 旁白总结：「用最玄学的极致概率，平替了让人崩溃的红黑树自旋变色机制，宏观平衡被完美控制」。

### 3.4 跳表 SkipList：【删】过河拆桥手法 (连线绕轨图)
- **图类型**：展示被移除时的连线动作。
- **绘图细节**：
    - 把阵型中间的某个自带 2 层高的节点标红打上“X”。
    - 让本该指向它的左侧高级指针纷纷画成弧形虚线，“高空跃过它”连线直达右侧继任节点。
    - 旁白：「大道至简：如同极简的单链表摘除，谁挡路就越过谁」。

### 3.5 跳表 SkipList：【改】偷步兵法的秘密 (重构决策流图)
- **图类型**：左右两路判断策略树。
- **绘图细节**：
    - **情况 A (只改内容，分数雷打不动)**：原地打钩，标注「既然没乱排名，原地修改完事」。
    - **情况 B (分数大变，排名剧变)**：画一个“垃圾桶（Delete）”把旧体销毁，并在远处的另一个排名山头抛下（Insert）一个新体。
    - 旁白大字凸显：**「Redis 面试绝杀坑点！只要分数一变，拒不原地挪动重排链表。绝对采用暴力“先全部按规矩删掉旧的，再从头按全新流程抛硬币 Insert 进队列！”」**

### 4. 字典 Dict 渐进式扩容 (2.3-字典.md) (双开状态图)
- **图类型**：扩容过渡期（精神分裂期）状态图。
- **主体元素**：
    - **左边 (老房子 `ht[0]`)**：装满了拉链法导致过长甚至拥挤串在一起的数据节点。
    - **右边 (新房子 `ht[1]`)**：一个容量为老房子两倍，但只装了零星几个转移数据的新数组。
- **动画流向 (搬砖逻辑)**：
    - 画一个小人或游标叫 `rehashidx`，正指向老房子的第一个格子的垃圾链表，标注「蚂蚁搬家：接客时顺路搬运一个桶」。
    - **新增警示牌**：画一个 `INSERT` 操作的新数据，加粗变红连线**一律只准进入新房子 `ht[1]`**（表示旧表只减不增）。

### 5. 全景架构：八大数据结构物理映射图 (2.4-全景物理映射.md) (拓扑映射树)
- **图类型**：左宏观（逻辑 API），右微观（底层物理池），形成 8对 N 的壮观网状线。
- **上路（基础武器库）**：
    - `String` -> 对应 `SDS`。
    - `List` -> 对应 `QuickList` (里面用花括号包裹 `ListPack`)。
    - `Set` -> 两条分岔：画个 `If: 全是数字` -> 对应 `IntSet`；否则进 `Dict`。
    - `ZSet (海量)` -> 罕见的汇聚线，一头连着 `Dict` (写：精准查分)，一头连着 `SkipList` (写：算排位找前100)。注：两条线最终合并指向同一块小积木（标：共享内存）。
- **下路（白嫖黑科技）**：
    - `BitMap` 与 `HyperLogLog` -> 汇聚一条线全部连回给老大哥 `SDS (String)`。标注：「挂羊头卖狗肉：巧用位运算和数学桶统计」。
    - `Stream` -> 连向 `Radix Tree (Rax)` 树干，而该树的所有树叶全部画成一截截的 `ListPack`（大巴车缝合体）。

## 3. 持久化机制 (3-redis持久化.md)

### 1. RDB: Fork 与 写时复制 (COW) (流程图)
- **图类型**：主进程与子进程分叉示意。
- **流程**：
    - `Main Process (Redis)` 执行 `fork()`。
    - 产生 `Child Process`。
    - **关键**：画一个内存页，主进程修改时产生副本 (Copy)，子进程读取原始页写入 `.rdb`。
    - **标注**：「不阻塞主线程」、「快照镜像」。

### 2. AOF 重写流程图（建议直接按流程图画）
- **流程主线（自上而下）**：
    - `开始重写（BGREWRITEAOF）`
    - `fork 子进程`
    - `子进程按内存快照写 New AOF.tmp`
    - `子进程完成通知主进程`
    - `主进程追加 AOF Rewrite Buffer 到 New AOF.tmp`
    - `atomic rename：New AOF.tmp -> New AOF`
    - `结束`
- **并行分支（挂在 fork 后，画为侧边分支）**：
    - `主进程继续处理写请求`
    - `每条新命令双写：AOF Buffer + AOF Rewrite Buffer`
    - `AOF Buffer -> write -> Page Cache -> fsync/回写 -> Old AOF`
- **子进程写盘链路（挂在“写 New AOF.tmp”节点下）**：
    - `Child Process -> write -> Page Cache -> fsync/回写 -> New AOF.tmp`
- **怎么连线（箭头文案可直接写）**：
    - `Client Write Command -> AOF Buffer`：`每条写命令`
    - `Client Write Command -> AOF Rewrite Buffer`：`重写期间增量命令`
    - `AOF Buffer -> Page Cache`：`write old AOF`
    - `Page Cache -> Old AOF`：`fsync / 回写`
    - `Child Process -> Page Cache`：`write new AOF.tmp`
    - `Page Cache -> New AOF.tmp`：`fsync / 回写`
    - `AOF Rewrite Buffer -> New AOF.tmp`：`append tail commands`
    - `New AOF.tmp -> New AOF (after rename)`：`atomic rename`
- **步骤文案（图旁注释）**：
    - `1) fork 后，子进程按内存快照重写新 AOF 主体`
    - `2) 主进程继续处理请求，并双写 AOF Buffer + Rewrite Buffer`
    - `3) 所有写文件先到 Page Cache，再按策略落盘`
    - `4) 子进程结束后，主进程补写 Rewrite Buffer 到新文件尾部`
    - `5) rename 原子替换旧 AOF，完成切换`
- **视觉区分建议**：
    - 普通写路径用蓝色实线，重写补尾用橙色虚线，最终替换用绿色粗线。
    - 在 `AOF Rewrite Buffer` 旁加注：`只在重写窗口期存在`。

## 5. 缓存进阶与一致性 (5-redis进阶.md)

### 1. 缓存三兄弟：穿透、击穿、雪崩 (对比矩阵)
- **图类型**：三栏卡片。
- **穿透**：库中无。方案：布隆过滤器、缓存空值。
- **击穿**：单点热 key 过期。方案：互斥锁、逻辑永不过期。
- **雪崩**：大面积 key 过期或服务宕机。方案：随机 TTL、集群高可用。

### 2. 布隆过滤器原理 (原理图)
- **元素**：`BitMap (0/1)`，多个 `Hash Functions`。
- **逻辑**：输入 Key -> 多哈希 -> 标记 BitMap 位。
- **结论**：全 1 可能在（误报），有 0 必不在。

### 3. 双写一致性：四种方案对比 (对比图/异常流程图)

- **图类型**：四格卡片，每格代表一种尝试方案及为什么“不行”。
- **内容**：
    1. **先更缓存，再更库**：标注「DB 失败 -> 缓存脏数据」。
    2. **先更库，再更缓存**：画 A、B 两个请求。A 先写库，B 后写库，B 先写缓存，A 后写缓存。标注「并发导致缓存覆盖」。
    3. **先删缓存，再更库**：画「请求 A 删缓存」->「请求 B 读库旧值填回缓存」->「请求 A 改库」。标注「瞬间回填旧值」。
    4. **先更库，再删缓存**：标准 Cache Aside 流程。标注「概率最低，极致要求加延迟双删」。

### 4. 极致一致性：重试与解耦 (关系流图)

- **方案 A：MQ 重试**
    - `业务代码` -> `写入 MQ` -> `消费者监听` -> `删除 Redis`。
    - 标注：「保证最终一致性，减少业务耦合」。
- **方案 B：订阅 Binlog (Canal)**
    - `MySQL` -> `Binlog` -> `Canal` -> `Redis`。
    - 标注：「业务零侵入，自动同步」。

### 5. 自动挡：Through 系列模式 (对比图)

- **图类型**：横向三栏。
- **Read Through**：标注「缓存自查，业务无感」。
- **Write Through**：标注「同步写库，数据一致性高」。
- **Write Behind**：标注「异步批量写库，吞吐量极致，有丢数据风险」。
- **设计感**：画出「业务层」、「缓存服务层」、「数据库层」的三层分层结构。
