## 有做什么优化吗
我先用 pprof 看接口热点，发现 GetDuplicateGroupList 这个链路耗时偏高。进一步看调用栈和 SQL 次数后，定位到典型 N+1 查询：先查分页分组列表 1 次，然后对每个 group 再单独查一次重复照片，导致总查询变成 1+N。
我再回到代码逻辑确认，原本循环里会按 group_id 逐组查明细。优化后改成先收集当前页所有 group_id，再一次 IN 批量查出所有照片，在内存里按组归并。这样 DB 往返从 1+N 降到 2 次，接口耗时明显下降。

## 如果让你重来一次，你会改什么？
如果重来一次，我会从开始起就引入标准的 Go 项目布局。在 LincAlbum 早期开发时，我更多关注功能的实现，导致目录结构混乱，功能模块分散，不利于维护和扩展。
比如utils全是杂七杂八的工具函数，没有按功能模块分类，导致后来很难找到需要的工具函数。比如早期 api 目录下混杂了 Controller 和 Service 逻辑，现在会建立清晰的请求流向：Handler → Service → Repository → Model

## 怎么保证上线质量？ 
自测 / 联调、Code Review、测试环境或预发跑一遍、关键路径有日志和监控、重要变更有回滚或开关、接口有基本校验等

## 线上出过事故吗？
相册算法端有时会出现 SIGABRT，也就是常见的 signal 6。排查到问题在于一个数组资源没有正确释放，导致状态不一。

### 你如何排查问题的？
根据问题类型，采取不同的排查方法
1. 功能不符合预期
    - 通过日志溯源，查看是否数据有偏移
    - 单元测试复现
2. 性能问题
    - 通过 pprof 分析热点函数，优化代码
