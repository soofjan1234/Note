## 为什么要加密

如果不加密，敏感字段会直接裸奔在链路中：

- NAS 设备 -> 云端：`给我发个访问码`
- 云端 -> NAS：`{accessID: "ABC123", accessVerify: "XYZ789", peerID: "peer001"}`

当前链路可以拆成两层：

1. **应用层加密**：设备侧先做 AES+RSA（混合加密）。
2. **传输层加密**：外层走 TLS（例如：`设备 -> [公网 TLS] -> Cloudflare -> http -> 云端`）。

若在 VPN 场景，常见链路是：`设备 -> [VPN 隧道] -> VPN 节点 -> [公网 TLS] -> Cloudflare`。

有些公司或特殊网络会要求在设备上安装其自有根证书（Root CA）。此时需注意：

- 一旦你信任了该根证书，VPN/代理节点理论上可以做“解密后再加密”（TLS 卸载 / 中间人代理）。
- 也就是说，**TLS 不再等同于“端到端只有你和服务端可见”**。

这时应用层 AES+RSA 的价值就体现出来了：

- 即便 TLS 在中间节点被解开，中间人看到的仍是应用层密文。
- 前提是：对端私钥和会话密钥管理得当，且加密实现没有漏洞。


## 加密方式有哪些

### 1. 对称加密

用同一把钥匙开同一把锁，AES

1. 速度快(比RSA快100-1000倍)
2. 密钥分发困难，怎么安全的把密钥传给对方

### 2. 非对称加密

有两把钥匙，一把公钥，一把私钥。用公钥锁上的东西，只有对应的私钥能打开，RSA

1. 密钥分发简单
2. 速度慢、只能加密小数据(RSA-2048最多245字节)

### 3. 哈希算法

1. 正向极快，逆向极难：适合数据库存密码，用户一输入就知道对不对
2. 差之毫厘，谬以千里：适合文件校验，下载文件后对比哈希值

### 4. 数字签名

私钥加密，公钥解密

1. 消息哈希值 + 私钥加密 = 签名
2. 公钥解密签名,对比哈希值
3. 身份认证、防篡改、不可抵赖

### 5. 混合加密

1. RSA传输AES密钥,AES加密实际数据
2. 既快又安全

## 混合加密的方法

1. **生成密钥**：客户端生成 AES 的 `key` 和 `nonce`
2. **加密数据**：客户端用 AES 加密 `data`
3. **加密密钥**：客户端用 RSA 公钥加密 AES 的 `key`
4. **解密密钥**：服务端拿 RSA 私钥解密 `key`
5. **还原数据**：然后还原 `data`
