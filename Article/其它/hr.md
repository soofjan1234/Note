## 动机
1. **为什么想跳槽**
我们公司每周有三天要加到8 9点，不管忙不忙，我不认同强制性加班
我认为比如发版，赶进度，有东西要弄懂解决的这类加班是合适的，而不是天天加班   
我希望加入一个节奏更科学、以结果为导向的团队

2. **什么是“节奏更科学、以结果为导向”的团队？**
- 目标明确化：任务有清晰的优先级和截止日期，而不是模棱两可的“全都要”。
- 反馈闭环：关注代码的质量和系统上线的稳定性，而不是坐在工位上的时长。
- 流程规范：有成熟的 CI/CD、自动化测试和发版流程，通过流程优化工作节奏。

3. **你如何判断我们公司就是这样的团队？**
- 面试的专业度：面试官问的技术问题是否深入，这代表了团队的技术追求。
- 反向提问：我会通过询问团队的“需求评审流程”和“版本迭代周期”来判断。

4. **未来的职业规划**
初期 快速熟悉公司业务 把现有项目高质量交付 同时和团队打好关系 熟悉合作模式
中期 根据公司业务深耕技术方向，比如高并发、分布式
后期 是走技术+管理路线 在保持技术深度的前提下，带团队、做项目管理，

## 行为与软素质（STAR：情境-任务-行动-结果）
1. **印象最深的一个项目？你在其中的角色？**
文档中心：我不仅负责了技术选型，也解决了一些难题。当时出现解析文件时内存占用高的问题，而且GPT在那时还不太聪明，可能会给出错误答案，我通过日志定位到是Bleve在索引阶段，然后去搜查解决方案，阅读源码。设计了分块、批量、限流的三层方案，最终解决了问题。

2. **说一次你推动别人或跨部门协作的经历。**
我们公司之前一直使用 Swagger 来管理 API。但我发现了apifox工具，可以先做接口设计以及讨论，再进行同步开发，这样能避免很多返工，同时也能提高开发效率。

3. **说一次你犯过的错或失败，后来怎么改的。**
在一个项目中，为了减少对下游接口的重复请求，设计了一个map来做缓存。但是只考虑了写入，却完全忽略了清理，后续map越来越大，导致内存溢出
所以现在会注意这个方面，比如用缓存时是否带容量限制、过期时间、定时清理。

4. **高压或 deadline 很紧时你怎么安排？**
评估压力的核心来源：是需求范围太大？是交付时间太短？还是存在外部技术依赖？
会定义MVP，哪些必须要做，哪些可以缓
我会把进度缩短到以‘半天’为单位。如果发现某个技术卡点可能导致延期，我会第一时间上升风险，而不是最后一天才说。
申请资源支持：如果判断人力确实不足，我会带着明确的任务分工向 Leader 申请资源

5. **缺点**

- 对于bug不分级就修改，现在我会根据重要度、紧急程度来修改bug
- 缺乏文档等等沉淀 以前习惯默默解决问题，现在会主动将解决方案和技术思考记录到 wiki，通过技术分享帮助团队，也方便后续查阅。

6. **一个需求下来 要展示要后端 你们怎么沟通的**

我们这边产品是技术出身，懂得比较多，所以需求阶段就能和技术对齐可行性，不会出现“产品拍板了技术实现不了”的情况。整体是**需求先对齐、再契约、再并行开发**：

**需求与方案阶段**
1. **方向提出**：产品提需求方向或目标（如“要做共享/外网访问”），会带一些技术上的想法。
2. **技术预研/对齐**：技术评估可行性（系统是否支持、有没有坑），因为产品懂技术，沟通成本低，预研结论能快速达成一致。
3. **线稿/UI 评审**：产品出线稿或高保真，技术+产品一起过交互和边界情况，细节定稿后再进入开发。

**开发与联调阶段**
4. **定义契约**：需求确认后，先出 API 文档（或接口约定），作为前后端共识。
5. **接口评审**：拉上前端/客户端确认字段是否够用，避免返工；产品有时也会参与，能听懂接口设计。
6. **并行开发**：提供 Mock 数据，前端不用等后端写完就能开工。
7. **高效联调**：最后联调；因为前面对齐得比较细，返工少。
