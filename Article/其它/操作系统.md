## 进程

### 进程的工作目录（CWD）

**它是什么**

每个进程在操作系统里都有一个**当前工作目录**（Current Working Directory，常简称 CWD 或 cwd）。

当你用**相对路径**打开文件或子路径时（例如 `./config.yaml`、`../logs/out.log`、或代码里的 `open("data.txt")`），系统会理解为：**相对于这个进程的 CWD 去找**，而不是相对于「可执行文件躺在磁盘上的那个目录」。

所以：**CWD ≠ 程序安装目录 / 二进制所在目录**，除非碰巧一致或启动时被人设成那样。

**那工作目录到底「在哪」**

不是全系统共用一个叫「工作目录」的固定文件夹，而是：**每个进程各自有一个当前目录**，内核为这个进程记着一条**绝对路径**（在 Linux 上本质是某个目录 inode 的引用）。

对你来说「在哪」= **看具体是哪个进程**，再问**它此刻 CWD 指向磁盘上的哪条路径**。例如可能是 `/home/alice/app`、`/tmp`、容器里的 `/app` 等，完全取决于：**谁启动的、启动前有没有 `cd`、有没有 `WorkingDirectory`/`WORKDIR`、程序里有没有 `chdir`**。

在 Linux 上查**真实路径**：`readlink -f /proc/<PID>/cwd`（或 `ls -l /proc/<PID>/cwd`）。这就是该进程当前「站在」的目录。

**谁会变、谁不会变**

- 操作系统给**每个进程单独记一份** CWD；进程 A 改了 CWD，**进程 B 完全不受影响**。
- 在 C 里叫 **`chdir()`**，在别的语言里往往是 `os.Chdir` 之类；**只改调用它的那个进程**。
- 启动子进程时：子进程刚诞生时，CWD **一般和父进程当时一样**（复制父进程环境）。子进程里若再 `exec` 成另一个程序，**CWD 通常仍然保持**，除非新程序一启动就自己 `chdir`，或由启动器（shell、systemd、容器）在 exec 之前就指定了目录。

**Shell 里 `cd` 为啥经常让人晕**

`cd` 是 **shell 内置命令**，作用只有一件事：**改当前这个 shell 进程自己的 CWD**。

因此：

- 你在终端里 `cd /tmp` 再 `./myapp`，`myapp` 作为子进程启动时，会继承 shell 已经变成 `/tmp` 的 CWD，所以相对路径相对的是 `/tmp`。这是**常见正确用法**。
- 若你写 `cd /tmp & ./myapp`（并行）：`&` 只让 `cd` 进后台；**`./myapp` 这条命令和 `cd` 不一定在同一个「先 cd 再执行」的顺序里**，实际 CWD 可能仍是原来的目录。可靠写法是 **`(cd /tmp && ./myapp)`** 用子 shell：先在一个**临时 shell 进程**里 `cd`，再执行，不影响外层 shell 的 CWD。
- 已经跑着的后台任务**不会**因为你后来在同一个终端里 `cd` 了而自动换目录；每个进程只认自己的那一份 CWD。

**脚本里想「到某目录再干活」**

任选一种清晰写法即可，例如：

```bash
(cd /path/to/proj && ./build.sh)   # 子 shell：只影响括号里的命令
# 或
cd /path/to/proj || exit 1
./build.sh                          # 当前脚本进程已切目录
```

能写**绝对路径**时最不容易错。

**在 Linux 上看某个进程的 CWD**

```bash
ls -l /proc/<PID>/cwd    # 符号链接指向该进程当前目录
readlink /proc/self/cwd  # 看当前 shell 自身
```

**服务 / 容器里谁定初始目录**

- **systemd**：单元文件里 **`WorkingDirectory=`**，表示这个服务主进程起来时的 CWD。
- **Dockerfile**：**`WORKDIR`**，后面 `RUN`/`CMD` 的相对路径都相对这里；也影响容器主进程的默认 CWD。

记住一条：**相对路径永远相对「这个进程自己的 CWD」**；要搞清行为，就查清**该进程启动时 CWD 被设成了什么**。

### 进程间通信

不同进程地址空间隔离，要交换数据或同步时，需经内核或双方约定的媒介。常见几类：

| 方式 | 要点 |
| --- | --- |
| **管道 Pipe** | 匿名管道，典型用于**父子进程**；单向字节流；随进程结束销毁。 |
| **命名管道 FIFO** | 文件系统上有路径，**无亲缘**进程也可打开读写；仍是字节流语义。 |
| **消息队列** | System V 或 POSIX `mq_*`；按消息块传递，可带类型/优先级（依实现）。 |
| **共享内存** | 多进程映射同一块物理页，**最快**，需**自配同步**（信号量、锁等），否则竞态。 |
| **信号量 / 文件锁** | 同步与互斥，常和共享内存、文件搭配。 |
| **信号 Signal** | 如 `SIGTERM`、`SIGUSR1`；信息量小，多用于通知、控制生命周期。 |
| **套接字 Socket** | 本机 **Unix 域套接字** 或 **网络 TCP/UDP**；最通用，可跨机（网络时）。 |
| **内存映射文件 `mmap`** | 把文件映射进地址空间；多进程映射同一文件可实现共享数据，同样要注意同步。 |
| **普通文件** | 写临时文件、日志、数据库等，也算广义的 IPC，简单但需约定格式与锁。 |

**选型粗记**：亲缘、单向、简单 → **pipe**；本机任意两进程、流式 → **FIFO / Unix socket**；大块、低延迟 → **共享内存 + 同步原语**；跨机或以后要扩展 → **socket**。
