网络分层与总览

一句话：网络就是分层协作，把数据从一个进程可靠/高效地送到另一个进程。

1) OSI vs TCP/IP（重点 TCP/IP）
- OSI 七层是理论模型，TCP/IP 四层是工程实践。
- 记忆顺序（自上而下）：应用层 -> 传输层 -> 网络层 -> 网络接口层。
- 常见协议：
  - 应用层：HTTP、HTTPS、DNS、WebSocket
  - 传输层：TCP、UDP
  - 网络层：IP、ICMP、ARP
  - 网络接口层：以太网、Wi-Fi

2) 输入网址会发生什么
- 1. URL 解析
  - 浏览器解析协议、域名、端口、路径（如 `https://example.com:443/path`）。
  - 命中 HSTS 时会强制走 HTTPS。
- 2. DNS 解析
  - 先查浏览器缓存、系统缓存、hosts 文件，再问本地 DNS。
  - 本地 DNS 递归/迭代查询到权威 DNS，返回 IP（可能是 CDN 就近节点）。
- 3. 建立连接
  - 先进行 TCP 三次握手。
  - HTTPS 还要 TLS 握手：验证证书、协商加密套件、生成会话密钥。
- 4. 发送请求
  - 浏览器组装 HTTP 请求行、请求头、Cookie、请求体并发送。
  - 服务端经过负载均衡 -> 网关/反向代理 -> 应用服务 -> 数据库/缓存，生成响应。
- 5. 返回响应
  - 服务端返回状态码、响应头、响应体（可能 gzip/br 压缩、分块传输）。
  - 浏览器按缓存策略处理（强缓存/协商缓存）。
- 6. 页面渲染
  - 解析 HTML 构建 DOM，解析 CSS 构建 CSSOM，合并为渲染树并布局绘制。
  - 遇到外链资源（JS/CSS/图片/字体）会继续发起请求；JS 可能阻塞渲染。
- 7. 连接管理
  - HTTP/1.1 默认 Keep-Alive，可复用连接。
  - 空闲超时后连接关闭；HTTP/2 多路复用可在单连接并发多个请求。

> HSTS: HTTP Strict Transport Security。告诉浏览器“这个站点以后只能用 HTTPS 访问，禁止再走 HTTP”。

传输层核心：TCP/UDP

1) TCP
- 面向连接、可靠传输、有序到达、全双工。
- 三次握手目的：
  - 双方确认收发能力正常
  - 初始化序列号
  - 避免历史连接干扰
- 四次挥手目的：双方分别关闭发送方向。
- 常见状态：
  - TIME_WAIT：主动关闭方等待 2MSL，确保最后 ACK 可重传、旧报文过期。
  - CLOSE_WAIT：被动关闭方收到 FIN 后等待应用层 close，时间过长通常是代码未及时关闭连接。

2) TCP 可靠性机制
- 序号 + ACK：确认哪些数据已收到。
- 超时重传：超时未确认就重发。
- 快速重传：连续收到重复 ACK，提前重传丢失段。
- 滑动窗口：提升吞吐，不必每发一包等一个 ACK。
- 流量控制：接收窗口 rwnd 防止接收方被打爆。
- 拥塞控制：cwnd + 慢启动/拥塞避免，防止网络被打爆。

3) UDP
- 无连接、尽力而为、不保证到达与顺序，头部开销小、延迟低。
- 适用：实时音视频、直播、游戏、高频小包场景。
- 可靠 UDP 一般在应用层自己实现（重传、排序、纠错）。

4) 什么时候选 TCP / UDP
- 选 TCP：文件传输、网页请求、支付/订单等强一致业务。
- 选 UDP：实时性优先，允许少量丢包（音视频、语音通话、游戏状态同步）。

网络层与路由

1) IP / 子网 / 网关
- IP 是主机在网络中的地址。
- 子网掩码/CIDR 用来划分网络号和主机号。
- 网关是“去其他网段”的下一跳。

2) ARP / ICMP / NAT
- ARP：已知 IP 找 MAC（局域网内通信必经步骤）。
- ICMP：网络控制与诊断（`ping` 基于 ICMP）。
- NAT：私网地址复用公网地址，常见在路由器出口。

3) 路由转发逻辑
- 路由表匹配核心是最长前缀匹配：匹配最具体的那条路由。

应用层高频协议

1) HTTP/1.1、HTTP/2、HTTP/3
- HTTP/1.1：文本协议，常用长连接；同一连接并发弱，易队头阻塞（应用层）。
- HTTP/2：二进制分帧 + 多路复用 + 头部压缩，性能明显更好。
- HTTP/3：基于 QUIC（UDP），减少握手时延，更好应对弱网和丢包。

2) HTTPS / TLS（简化）
- 服务端用证书证明身份（证书链 + CA 信任）。
- 握手阶段协商对称密钥。
- 后续业务数据用对称加密传输（快），并做完整性校验。

3) DNS 查询过程
- 浏览器缓存 -> 系统缓存 -> 本地 DNS 服务器。
- 本地 DNS 递归查询，上游通常迭代到根/顶级域/权威 DNS。
- 结果按 TTL 缓存。

4) WebSocket
- 先通过 HTTP Upgrade 建连，再升级为全双工长连接。
- 适合实时推送：IM、行情、协同编辑、通知系统。

性能与优化

1) 指标
- 延迟：一次请求耗时。
- 吞吐量：单位时间处理的数据量/请求量。
- RTT：往返时延，是很多网络性能问题的基础变量。

2) 常用优化手段
- 长连接 + 连接池：减少频繁建连成本。
- Keep-Alive：提高连接复用率。
- 压缩（gzip/br）：减少传输字节。
- CDN：就近访问，降低跨地域延迟。
- 缓存：强缓存 + 协商缓存（ETag/Last-Modified）。

3) 常见问题
- 粘包/拆包：TCP 是字节流，不是消息边界；应用层要定义协议边界（长度字段或分隔符）。
- 队头阻塞：慢包影响后续包，HTTP/2/3 都在尽量缓解。

排障速查

排查顺序（建议固定模板）：
1. DNS 是否正常（域名能否解析）
2. TCP 是否可达（三次握手是否成功）
3. TLS 是否通过（证书/协议版本/加密套件）
4. HTTP 是否异常（状态码、超时、重试）

常用命令：
- `ping`：看连通性和延迟
- `tracert` / `traceroute`：看链路路径和中间跳点
- `nslookup` / `dig`：看 DNS 解析结果
- `netstat`：看端口监听和连接状态
- `curl -v`：看请求详细过程

高频面试快问快答

1) 为什么是三次握手，不是两次？
- 两次无法让双方都确认“对方的收发能力 + 自己的收发能力”都正常，也无法可靠同步初始序列号。

2) TIME_WAIT 为什么存在？
- 防止最后 ACK 丢失导致被动关闭方无法正常结束；同时让旧连接报文在网络中自然过期。

3) TCP 如何保证可靠传输？
- 序号、ACK、重传、窗口机制、流量控制、拥塞控制协同完成。

4) HTTP 和 HTTPS 区别？
- HTTPS = HTTP + TLS；提供身份认证、机密性、完整性保护。
